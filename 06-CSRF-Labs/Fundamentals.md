# Cross-Site Request Forgery (CSRF) Fundamentals

Cross-Site Request Forgery (CSRF) is a web security vulnerability that allows an attacker to induce users to perform actions that they do not intend to perform. It partially circumvents the Same-Origin Policy (SOP), which is designed to prevent different websites from interfering with each other.

```
Client-Side (Browser): Exploits the browser's default behavior of automatically attaching active session cookies when making cross-site requests.
Server-Side: Server processes the request blindly without verifying if the request originated from the authentic user interface.
```

---

## 3 Core Conditions for CSRF

A CSRF vulnerability requires three conditions to be met:
1. **A Relevant Action:** A state-changing action within the application (e.g. changing email/password, completing a checkout, modifying account settings).
2. **Cookie-Based Session Handling:** The application relies solely on HTTP cookies to validate the user's active session.
3. **No Unpredictable Parameters:** The HTTP request parameters required to perform the action are predictable, meaning they do not contain secret, dynamic values (such as an anti-CSRF token).

---

## The Role of Anti-CSRF Tokens

An anti-CSRF token is a unique, unpredictable, and cryptographically secure value generated by the server and bound to the user's active session.
- **Mechanism:** The server embeds the token in forms (e.g. `<input type="hidden" name="csrf" value="TOKEN">`) or header fields. When submitting data, the server validates the received token against the session store.
- **Why it stops attacks:** Because of Same-Origin Policy (SOP), an external attacker's website (`evil.com`) cannot read the token value from the target domain's response, rendering them unable to forge a valid request.

---

## SameSite Cookie Restrictions

`SameSite` is a cookie attribute that tells the browser whether to attach cookies in cross-site contexts.

1. **`SameSite=Strict`:** The browser **never** sends the cookie in cross-site requests (e.g., clicking a link to the site from a search engine will show the user as logged out).
2. **`SameSite=Lax`:** (Default browser behavior) Cookies are blocked on cross-site requests, **except** for top-level navigations using the `GET` method (e.g., standard links).
3. **`SameSite=None`:** Cookies are sent in all cross-site contexts, but must specify the `Secure` attribute (HTTPS only).

---

## Common CSRF Bypass Techniques

- **HTTP Method Swapping:** If the server checks tokens only on `POST` requests, swap the method to `GET` (e.g. `/email/change?email=attacker@evil.com`).
- **Token Absence Handling:** Removing the token parameter completely to see if the validation checker skips verification.
- **Unassociated Token Pool:** Using a valid token generated from the attacker's account to pass the validation check on the victim's session.
- **Cookie-to-Token Validation Discrepancy:** Injecting or modifying a custom cookie (using CRLF or search parameter injection) to match a spoofed token sent in the request body.
- **Referer Header Spoofing:** Circumventing validation by removing the `Referer` header using `<meta name="referrer" content="no-referrer">`, or exploiting regex flaws in referrer domain checks.
